# UNOUN — Integration & Launch Checklist

Этот документ фиксирует, что нужно подготовить заранее для боевого запуска интеграций и выгрузки проекта.

## 1. GitHub

Что уже видно по проекту сейчас:
- локальный git-репозиторий уже есть;
- активная ветка: `main`;
- remote `origin` уже подключен;
- первый `push` в GitHub уже выполнен;
- `gh` CLI в системе сейчас недоступен.

Что нужно от владельца проекта дальше:
1. Подтвердить, что `main` остается production-веткой.
2. Решить, куда будет деплоиться проект с GitHub:
   - `Timeweb VDS`
   - отдельный deploy user / SSH key
3. Подготовить доступы, если деплой будет настраиваться не вручную, а через CI/CD.

Что потом делаем в проекте:
1. Настраиваем deploy flow.
2. Привязываем production env.
3. После этого GitHub становится source of truth не только для кода, но и для релизов.

## 2. Yandex Pay / Split

Что нужно зарегистрировать уже сейчас:
1. Войти в `Яндекс ID`, который будет привязан к бизнесу, а не к случайному личному аккаунту.
2. Добавить организацию в кабинет `Яндекс Пэй для бизнеса`.
3. Подать заявку на подключение `Яндекс Пэй` и `Сплит`.

По официальным шагам Яндекс Пэй потребует:
- сайт магазина;
- название магазина;
- вид деятельности;
- ожидаемый оборот;
- контактное лицо;
- БИК;
- расчетный счет для выплат.

Что важно для проекта после одобрения:
- `merchantId`;
- API-ключ;
- доступ к тестовой среде;
- webhook / notification URL;
- возможность включить полную оплату и `Split`.

Что еще важно по сайту:
- сайт должен работать по `HTTPS`;
- CSP должна разрешать загрузку ресурсов с `https://pay.yandex.ru`;
- не должен мешать заголовок `Cross-Origin-Opener-Policy`.

Практический совет:
- даже до полной интеграции можно включить `оплату по ссылке` и прогнать первые тестовые сценарии;
- это полезно как временный мост, если backend-интеграция будет делаться позже.

## 3. Яндекс Доставка

Что нужно сделать уже сейчас:
1. Войти в личный кабинет `Яндекс Доставки`.
2. На вкладке `Интеграция` получить Bearer token.
3. Уточнить у менеджера боевой `platform_station_id` для точки отправки.
4. Получить `merchant_id` после регистрации мерчанта.

Для проекта понадобятся:
- `YANDEX_DELIVERY_API_TOKEN`;
- `YANDEX_DELIVERY_MERCHANT_ID`;
- `YANDEX_DELIVERY_SOURCE_STATION_ID`;
- тестовая среда;
- доступ к API ПВЗ;
- доступ к созданию офферов и заявок;
- механизм получения статусов.

Что должно получиться в MVP:
1. Пользователь вводит город.
2. Сайт получает список ПВЗ `Яндекс Доставки`.
3. Пользователь выбирает пункт выдачи.
4. Мы сохраняем код ПВЗ, адрес и внешний delivery context в заказ.
5. После оплаты подтверждаем оффер и создаем заявку в Яндекс Доставке.

Практический совет:
- сначала запускать только сценарий `ПВЗ -> ПВЗ`;
- тестировать ПВЗ в тестовом контуре нужно на Москве;
- сразу запросить у менеджера и боевой токен, и боевой `platform_station_id`.

## 4. Timeweb / сервер / база

Для следующего этапа подготовки нужны:
- доступ в `Timeweb`;
- VDS или подтверждение, что он будет куплен;
- Managed MySQL в РФ;
- SSH-доступ;
- домен, который будет указывать на сервер;
- почта, через которую будут приходить технические уведомления.

Переменные окружения под это уже подготовлены в `.env.example`.

## 5. Что можно делать без внешних доступов

Пока нет ключей и договоров, можно безопасно продолжать:
- строить server-side архитектуру заказа;
- готовить таблицы данных;
- делать auth-слой;
- строить admin MVP;
- усиливать SEO и метаданные;
- готовить интеграционные адаптеры под `Yandex Pay` и `Яндекс Доставку`.

## 6. Что нужно от владельца прямо сейчас

Минимальный practical checklist:
1. Зарегистрироваться в `Яндекс Пэй для бизнеса` и подать заявку на `Пэй + Сплит`.
2. Получить API-доступы в `Яндекс Доставке` и забрать токен из кабинета.
3. Подготовить данные для `.env`:
   - production domain;
   - support email;
   - legal email;
   - support phone.
4. Подтвердить инфраструктуру под backend:
   - `Timeweb VDS`
   - `Managed MySQL`
   - доступы для дальнейшего деплоя и миграций

Что уже сделано в проекте на текущем этапе:
- checkout больше не только UI: добавлен server-side route `app/api/checkout/route.ts`;
- подготовлена Prisma schema для `users`, `addresses`, `products`, `orders`, `order_items`, `bonus_accounts`, `bonus_transactions`, `articles`;
- добавлен `mock` / `database` режим для checkout, чтобы можно было двигаться без живой БД;
- проект проходит `lint` и `production build`.

После получения внешних доступов можно переходить к боевому database-режиму и реальным интеграциям.
